Implement `DhtClient` in `pktap-core/src/dht.rs`. The struct wraps `pkarr::PkarrClient` and provides PKTap-specific publish/resolve.

**Critical architecture (from 02-RESEARCH.md):** The DHT address is the signing keypair's Ed25519 public key. The `_pktap._share.<hash>` is a DNS record name WITHIN the signed packet, not the DHT address. Both peers derive the same `pkarr::Keypair` from their identical HKDF-derived shared secret (per D-02).

**Constants:**
```rust
/// Default TTL for private encrypted records (24 hours).
pub const PRIVATE_RECORD_TTL: u32 = 86_400;
/// Default TTL for public profile records (7 days).
pub const PUBLIC_RECORD_TTL: u32 = 604_800;
/// Maximum ciphertext size that fits in a pkarr DNS packet.
/// 1000 (BEP-44 limit) - 12 (DNS header) - ~90 (name + RR overhead + TXT length prefixes) = ~898.
/// Use 850 as conservative safe limit.
pub const MAX_CIPHERTEXT_LEN: usize = 850;
```

**DhtClient struct:**
```rust
pub struct DhtClient {
    inner: PkarrClient,
}
```

**Methods to implement:**

1. `DhtClient::new() -> Result<Self, PktapError>` — Creates PkarrClient with default settings (production DHT bootstrap nodes).

2. `DhtClient::with_bootstrap(bootstrap: Vec<String>) -> Result<Self, PktapError>` — Creates PkarrClient with custom bootstrap (for tests).

3. `DhtClient::publish_encrypted(&self, hkdf_derived_seed: &[u8; 32], record_name: &str, ciphertext: &[u8]) -> Result<pkarr::PublicKey, PktapError>` — Validates ciphertext size (reject if > MAX_CIPHERTEXT_LEN with RecordTooLarge). Builds `Keypair::from_secret_key(hkdf_derived_seed)`. Constructs DNS packet: `Packet::new_reply(0)`, TXT record with name `record_name`, TTL `PRIVATE_RECORD_TTL`, binary data split into 255-byte `CharacterString` chunks. Calls `SignedPacket::from_packet(&keypair, &packet)`, then `self.inner.publish(&signed_packet)`. Maps pkarr errors: `Error::NotMostRecent` -> `DhtOutdatedRecord`, network errors -> `DhtPublishFailed`. Returns the keypair's public key (the DHT address) on success.

4. `DhtClient::resolve_encrypted(&self, signer_public_key: &pkarr::PublicKey, record_name: &str) -> Result<Option<Vec<u8>>, PktapError>` — Calls `self.inner.resolve(signer_public_key)`. If `Some(signed_packet)`, iterates `signed_packet.resource_records(record_name)`, finds TXT record, reassembles binary from CharacterString chunks. Returns `Ok(Some(bytes))` or `Ok(None)` if not found. Maps errors to `DhtResolveFailed`.

5. `DhtClient::publish_public(&self, profile_seed: &[u8; 32], record_name: &str, plaintext: &[u8]) -> Result<pkarr::PublicKey, PktapError>` — Same as publish_encrypted but uses `PUBLIC_RECORD_TTL` (604800). Validates plaintext size. Returns DHT address.

6. `DhtClient::resolve_public(&self, signer_public_key: &pkarr::PublicKey, record_name: &str) -> Result<Option<Vec<u8>>, PktapError>` — Same as resolve_encrypted (same wire format).

**DNS packet construction helper (private):**

fn build_signed_packet(

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DhtClient -> Mainline DHT | Untrusted network; DHT nodes may return stale/malicious packets |
| Caller -> DhtClient | Caller provides HKDF seed bytes (trusted, from Phase 1 crypto) and ciphertext (trusted, from ecdh_and_encrypt) |
| DHT -> Resolver | Resolved packets must be verified via BEP-44 Ed25519 signature (pkarr handles this internally) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | DhtClient::resolve_encrypted | mitigate | pkarr verifies BEP-44 Ed25519 signature on every resolved SignedPacket internally; DhtClient does not bypass this |
| T-02-02 | Tampering | DNS packet in transit | mitigate | BEP-44 signatures cover the entire bencoded value; any bit flip invalidates the signature; pkarr rejects tampered packets |
| T-02-03 | Repudiation | DhtClient::publish_encrypted | accept | DHT publishes are signed but anonymous; the signing key is derived from shared ECDH secret, not identity; repudiation is a feature (privacy) |
| T-02-04 | Information Disclosure | HKDF seed passed to DhtClient | mitigate | Seed is used only inside `Keypair::from_secret_key()` which copies internally; DhtClient does not store the seed; caller zeros after use |
| T-02-05 | Denial of Service | Oversized record publish | mitigate | DhtClient pre-validates ciphertext size via MAX_CIPHERTEXT_LEN before building packet; rejects with RecordTooLarge |
| T-02-06 | Elevation of Privilege | DHT address derivation | accept | Both peers derive the same DHT address from ECDH shared secret; this is by design (symmetric key agreement); no privilege boundary exists |
| T-02-07 | Information Disclosure | DNS record name leaks peer relationship | accept | The `_pktap._share.<SHA-256(sort(A,B))>` hash within the packet reveals that two specific public keys exchanged data, but only to someone who already knows both public keys; low risk given keys are exchanged via NFC/QR (physical proximity) |
</threat_model>
