Extend `DhtClient` to include an offline queue per D-03 and D-04.

**Change DhtClient struct** to include mutable queue state. Since `PkarrClient` is internally thread-safe but DhtClient needs mutable queue access, wrap the queue in a `Mutex`:

```rust
use std::collections::VecDeque;
use std::sync::Mutex;
use std::time::{Duration, Instant};

struct PendingPublish {
    signed_packet: SignedPacket,
    next_attempt: Instant,
    attempt_count: u32,
}

pub struct DhtClient {
    inner: PkarrClient,
    queue: Mutex<VecDeque<PendingPublish>>,
}
```

**Update constructors** (`new()` and `with_bootstrap()`) to initialize the queue field: `queue: Mutex::new(VecDeque::new())`.

**Modify publish_encrypted and publish_public:** On success, return Ok as before. On network error from `self.inner.publish()`, enqueue a `PendingPublish` with `next_attempt: Instant::now() + Duration::from_secs(1)` and `attempt_count: 0`. Return `Ok(pubkey)` (the publish is queued, not failed — per D-04 the attempt IS the connectivity check). Alternatively, return `Err(PktapError::DhtPublishQueued)` to let the caller know it was queued rather than immediately published. Choose `DhtPublishQueued` as the return to give callers visibility.

**Add methods:**

1. `pub fn flush_queue(&self) -> usize` — Lock the queue mutex. Iterate items. For each item where `item.next_attempt <= Instant::now()`, attempt `self.inner.publish(&item.signed_packet)`. On success, remove from queue. On failure, increment `attempt_count`, compute `delay = Duration::from_secs((1u64 << attempt_count).min(60))`, set `next_attempt = Instant::now() + delay`, push to back of queue. Return number of items successfully flushed.

2. `pub fn queue_len(&self) -> usize` — Returns current queue length. Used by tests and by Phase 6 for "pending sync" indicator.

**Backoff formula per D-04:** `delay_secs = min(2^attempt_count, 60)`. So: attempt 0 -> 1s, attempt 1 -> 2s, attempt 2 -> 4s, ..., attempt 6+ -> 60s.

**Testing strategy:** For queue tests, create a DhtClient with an unreachable bootstrap (e.g., `vec!["127.0.0.1:1".to_string()]`) so publish always fails. Then verify the queue grows. For flush tests, you can inspect the PendingPublish items via queue_len(). For the full flush round-trip, use a testnet client.

Note: The `PendingPublish` struct stores the `SignedPacket` (which is `Clone` in pkarr 2.3.0). Verify that `SignedPacket` implements `Clone` — if not, use `Arc<SignedPacket>`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TrackedRecord seed storage | HKDF seed bytes held in memory for republishing; must be zeroed on drop |
| Public mode plaintext | Public records are intentionally unencrypted; the user explicitly opts in (Phase 7 PUB-01) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-08 | Information Disclosure | TrackedRecord.seed in memory | mitigate | Derive Zeroize/ZeroizeOnDrop on TrackedRecord; seed bytes zeroed when DhtClient is dropped |
| T-02-09 | Denial of Service | Unbounded offline queue growth | mitigate | Queue is VecDeque in memory; bounded by process lifetime (D-03: lost on process kill); Phase 6 can add a max queue size if needed |
| T-02-10 | Spoofing | Public mode records | accept | Public records are intentionally readable by anyone with the public key; no spoofing concern because BEP-44 signature verifies the publisher |
| T-02-11 | Tampering | Republish with stale data | mitigate | pkarr auto-generates new microsecond timestamp on republish; Error::NotMostRecent prevents overwriting newer records |
</threat_model>
